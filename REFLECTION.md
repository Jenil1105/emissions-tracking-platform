## What I Learned Using AI Agents

This assignment helped me understand that AI agents are most effective when used as **accelerators rather than decision-makers**. Initially, I was not very familiar with hexagonal architecture, so I implemented the backend and frontend in a straightforward REST-based structure without strict separation of concerns.

Using AI agents (primarily Codex), I was able to refactor the entire system into a proper hexagonal architecture. This process taught me how to **map real-world domain logic (CB, banking, pooling)** into structured layers such as core, ports, and adapters.

Another key learning was that AI agents are good at **code transformation and UI generation**, but they often lack deep understanding of **domain-specific rules**. For example, the agent generated incorrect logic for banking and pooling, which forced me to deeply understand the system and manually correct it. This significantly improved my understanding of the domain.

Overall, I learned that:

* AI is excellent for **refactoring, scaffolding, and UI work**
* But **domain logic and correctness must be validated manually**

---

## Efficiency Gains vs Manual Coding

The biggest efficiency gain came from **refactoring and UI development**.

* Refactoring an entire monolithic REST API into hexagonal architecture manually would have taken significant time. Codex accelerated this by restructuring files and layers quickly, allowing me to focus on correctness rather than boilerplate.
* In the frontend, generating layouts for multiple pages (Routes, Compare, Banking, Pooling) and styling them with Tailwind was much faster using AI. Tasks like spacing, responsiveness, and component structuring were handled efficiently.

However, for **core business logic**, manual work was still essential:

* Banking logic initially generated by the agent was incorrect (ship-to-ship instead of ship-year aggregation)
* Pooling logic did not consider banked adjustments
* There was no proper integration between banking and pooling flows

Fixing these required manual reasoning and debugging, where AI assistance was limited.

--> In summary:

* AI significantly reduced time in **boilerplate, UI, and restructuring**
* Manual effort was still required for **correct system design and domain logic**

## Improvements for Next Time

If I were to approach this assignment again, I would improve both my **development strategy** and **AI usage**:

### 1. Better Upfront Architecture Planning

Instead of building a monolithic version first and refactoring later, I would:

* Design the hexagonal architecture upfront
* Define clear boundaries between core, ports, and adapters before implementation

This would reduce rework and make AI usage more structured.

---

### 2. More Controlled Use of AI Agents

I would use AI more **incrementally and precisely**, such as:

* Generating only specific use cases or components instead of large blocks of code
* Avoiding over-reliance on AI for business logic

---

### 3. Better Integration Between System Components

Initially, banking and pooling were treated as separate features. After debugging, I realized the correct flow is:

```
Routes → CB → Banking Adjustment → Adjusted CB → Pooling
```

Next time, I would design this flow upfront to avoid inconsistencies.

---

## Final Reflection

This assignment highlighted that AI agents are powerful tools for **speed and productivity**, but they require **strong guidance and validation**. The combination of AI-assisted development and manual reasoning allowed me to build a structured system while also gaining a deeper understanding of both architecture and domain logic.
